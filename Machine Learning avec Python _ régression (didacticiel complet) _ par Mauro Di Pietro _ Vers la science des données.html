Analyse des données

En statistique, l’analyse exploratoire des données est le processus de synthèse des principales caractéristiques d’un ensemble de données pour comprendre ce que les données peuvent nous dire au-delà de la modélisation formelle ou de la tâche de test d’hypothèse.

Je commence toujours par obtenir un aperçu de l’ensemble de l’ensemble de l’ensemble, en particulier, je veux savoir combien de variables catégorielles et numériques il y a et la proportion de données manquantes. Reconnaître le type d’une variable peut parfois être difficile, car les catégories peuvent être exprimées sous forme de nombres. À cette fin, je vais écrire une fonction simple qui fera cela pour nous:

'''
Recognize whether a column is numerical or categorical.
:parameter
    :param dtf: dataframe - input data
    :param col: str - name of the column to analyze
    :param max_cat: num - max number of unique values to recognize a column as categorical
:return
    "cat" if the column is categorical or "num" otherwise
'''
def utils_recognize_type(dtf, col, max_cat=20):
    if (dtf[col].dtype == "O") | (dtf[col].nunique() < max_cat):
        return "cat"
    else:
        return "num"

Lorsque vous n’êtes pas convaincu par « l’intuition oculaire », vous pouvez toujours recourir à de bonnes vieilles statistiques et effectuer un test. Dans ce cas de catégoriel (FullBath) vs numérique (Y), j’utiliserais un test ANOVA one-way. Fondamentalement, il teste si les moyennes de deux ou plusieurs échantillons indépendants sont significativement différentes, donc si la valeur de p est suffisamment petite (<0,05), l’hypothèse nulle des échantillons signifie que l’égalité peut être rejetée.

cat, num = "FullBath", "Y"model = smf.ols(num+' ~ '+cat, data=dtf).fit()
table = sm.stats.anova_lm(model)
p = table["PR(>F)"][0]
coeff, p = None, round(p, 3)
conclusion = "Correlated" if p < 0.05 else "Non-Correlated"
print("Anova F: the variables are", conclusion, "(p-value: "+str(p)+")")

Nous pouvons conclure que le nombre de salles de bains détermine un prix plus élevé de la maison. Cela a du sens car plus de salles de bains signifient une maison plus grande et la taille de la maison est un facteur de prix important.

Ingénierie des fonctionnalités

Il est temps de créer de nouvelles fonctionnalités à partir de données brutes à l’aide de la connaissance du domaine. Je vais fournir un exemple: la colonne MSSubClass (la classe de construction) contient 15 catégories, ce qui est beaucoup et peut provoquer un problème de dimensionnalité lors de la modélisation. Voyons:

Je vais donc regrouper ces catégories en clusters: les classes avec une valeur Y plus élevée (comme MSSubClass 60 et 120)iront dans le cluster « max », les classes avec des prix plus bas (comme MSSubClass 30, 45, 180)iront dans le cluster « min », le reste sera regroupé dans le cluster « moyen ».

Prétraitement

Le prétraitement des données est la phase de préparation des données brutes pour les rendre adaptées à un modèle d’apprentissage automatique. En particulier:

  1. chaque observation doit être représentée par une seule ligne, en d’autres termes, vous ne pouvez pas avoir deux lignes décrivant le même passager, car elles seront traitées séparément par le modèle (le jeu de données est déjà sous cette forme, donc ✅). De plus, chaque colonne doit être une fonctionnalité, vous ne devez donc pas utiliser Id comme prédicteur, c’est pourquoi ce type de table est appelé «matrice d’entités».
  2. Le jeu de données doit être partitionné en au moins deux ensembles : le modèle doit être formé sur une partie importante de votre jeu de données (appelé « ensemble de données ») et testé sur un ensemble plus petit (« ensemble de tests »).
  3. Les valeurs manquantes doivent être remplacées par quelque chose, sinon votre modèle peut paniquer.
  4. Les données catégorielles doivent être encodées, ce qui signifie convertir des étiquettes en entiers, car l’apprentissage automatique attend des nombres, et non des chaînes.
  5. Il est recommandé de mettre à l’échelle les données, cela permet de normaliser les données dans une plage particulière et d’accélérer les calculs dans un algorithme.

D’accord, commençons par partitionner le jeu de données. Lorsque vous fractionnerez des données en trains et en ensembles de test, vous devez suivre 1 règle de base : les lignes de l’ensemble de tests ne doivent pas apparaître également dans l’ensemble de tests. En effet, le modèle voit les valeurs cibles pendant l’apprentissage et l’utilise pour comprendre le phénomène. En d’autres termes, le modèle connaît déjà la bonne réponse pour les observations de formation et le tester sur celles-ci serait comme tricher.

Sélection des fonctionnalités

La sélection des fonctionnalités est le processus de sélection d’un sous-ensemble de variables pertinentes pour générer le modèle d’apprentissage automatique. Il facilite l’interprétation du modèle et réduit le surajustement (lorsque le modèle s’adapte trop aux données d’apprentissage et fonctionne mal en dehors de la rame).

J’ai déjà fait une première sélection de fonctionnalité « manuelle » lors de l’analyse des données en excluant les colonnes non pertinentes. Maintenant, ça va être un peu différent parce que nous devons faire face au problème de multicolinéarité, qui fait référence à une situation dans laquelle deux ou plusieurs variables explicatives dans un modèle de régression multiple sont fortement liées linéairement.

Je vais expliquer avec un exemple: GarageCars est fortement corrélé avec GarageArea parce qu’ils donnent tous les deux les mêmes informations (quelle est la taille du garage, l’un en termes de combien de voitures peuvent s’adapter, l’autre en pieds carrés). Calculons la matrice de corrélation pour la voir :

corr_matrix = dtf_train.corr(method="pearson")
sns.heatmap(corr_matrix, vmin=-1., vmax=1., annot=True, fmt='.2f', cmap="YlGnBu", cbar=True, linewidths=0.5)
plt.title("pearson correlation")

Un parmi garagecars et GarageArea pourrait être inutile et nous pouvons décider de le laisser tomber et de garder le plus utile (c’est-à-dire celui avec la valeur de p la plus faible ou celui qui réduit le plus l’entropie).

La régression linéaire est une approche linéaire de modélisation de la relation entre une réponse scalaire et une ou plusieurs variables explicatives. Les tests de régression linéaire univariée sont largement utilisés pour tester l’effet individuel de chacun des nombreux régresseurs: tout d’abord, la corrélation entre chaque régresseur et la cible est calculée, puis un test F ANOVA est effectué.

La régularisation RIDGE est particulièrement utile pour atténuer le problème de la multicolinéarité dans la régression linéaire, qui se produit généralement dans les modèles avec un grand nombre de paramètres.

X = dtf_train.drop("Y", axis=1).values

               feature_selection.f_regression, k=10).fit(X,y)

              linear_model.Ridge(alpha=1.0, fit_intercept=True),

## plot

dtf_features["regularization"] = dtf_features["features"].apply(lambda x: "regularization" if x in regularization_selected_features else "")

Conception du modèle

Enfin, il est temps de créer le modèle d’apprentissage automatique. Je vais d’abord exécuter une régression linéaire simple et l’utiliser comme base de référence pour un modèle plus complexe, comme l’algorithme d’amplification de gradient.

La première métrique que j’utilise normalement est le R au carré, qui indique la proportion de la variance dans la variable dépendante qui est prévisible à partir de la variable indépendante.

Je vais comparer la régression linéaire R au carré avec celle du gradient boosting en utilisant la validation croisée k-fold, une procédure qui consiste à diviser les données k fois en ensembles de train et de validation et pour chaque fractionnement, le modèle est formé et testé. Il est utilisé pour vérifier dans quelle mesure le modèle est capable d’être formé par certaines données et de prédire des données invisibles.

Je vais visualiser les résultats de la validation en traçant les valeurs prédites par rapport au Yréel. Idéalement, les points doivent être tous proches d’une ligne diagonale où prédit = réel.

## call model
model = linear_model.LinearRegression()## K fold validation

fig = plt.figure()

for train, test in cv.split(X_train, y_train):
    prediction = model.fit(X_train[train],
                 y_train[train]).predict(X_train[test])
    true = y_train[test]
    score = metrics.r2_score(true, prediction)

                label='Fold %d (R2 = %0.2f)' % (i,score))

Évaluation

Moment de vérité, nous sommes sur le point de voir si tout ce travail acharné en vaut la peine. Le but est d’étudier la variance de Y que le modèle peut expliquer et comment les erreurs sont distribuées.

Je vais évaluer le modèle à l’aide des métriques courantes suivantes : R au carré, erreur absolue moyenne (MAE)et erreur quadratique moyenne (RMSD). Les deux derniers sont des mesures d’erreur entre des observations appariées exprimant le même phénomène. Étant donné que les erreurs peuvent être à la fois positives (prédiction > réelle) et négatives (prédiction < réelle), vous pouvez mesurer la valeur absolue et la valeur au carré de chaque erreur.

## Kpi
print("R2 (explained variance):", round(metrics.r2_score(y_test, predicted), 2))
print("Mean Absolute Perc Error (Σ(|y-pred|/y)/n):", round(np.mean(np.abs((y_test-predicted)/predicted)), 2))
print("Mean Absolute Error (Σ|y-pred|/n):", "{:,.0f}".format(metrics.mean_absolute_error(y_test, predicted)))
print("Root Mean Squared Error (sqrt(Σ(y-pred)^2/n)):", "{:,.0f}".format(np.sqrt(metrics.mean_squared_error(y_test, predicted))))## residuals
residuals = y_test - predicted
max_error = max(residuals) if abs(max(residuals)) > abs(min(residuals)) else min(residuals)
max_idx = list(residuals).index(max(residuals)) if abs(max(residuals)) > abs(min(residuals)) else list(residuals).index(min(residuals))
max_true, max_pred = y_test[max_idx], predicted[max_idx]
print("Max Error:", "{:,.0f}".format(max_error))

Le modèle explique 86 % de la variance de la variable cible. En moyenne, les prédictions ont une erreur de 20k $, ou elles sont erronées de 11 %. La plus grande erreur sur l’ensemble de test était de plus de 170k $. Nous pouvons visualiser les erreurs en traçant prédit par rapport aux réels et le résiduel (l’erreur) de chaque prédiction.

Explicabilité

Vous avez analysé et compris les données, vous avez formé un modèle et l’avez testé, vous êtes même satisfait des performances. Vous pouvez faire un effort supplémentaire et montrer que votre modèle d’apprentissage automatique n’est pas une boîte noire.

Le paquet Lime peut nous aider à construire un explicatif. Pour donner une illustration, je vais prendre une observation aléatoire de l’ensemble de test et voir ce que le modèle prédit:

Le modèle prévoyait un prix pour cette maison de 194 870 $. Pourquoi? Utilisons l’explicatif:

Les principaux facteurs de cette prédiction particulière sont que la maison a un grand sous-sol (TotalBsmft > 1.3k), il a été construit avec des matériaux de haute qualité (OverallQual > 6), et il a été construit récemment (YearBuilt > 2001).

Le graphique prédit par rapport aux chiffres réels est un excellent outil pour montrer comment les tests se sont déroulés, mais je trace également le plan de régression pour donner une aide visuelle des observations aberrantes que le modèle n’a pas prédites correctement. Comme cela fonctionne mieux pour les modèles linéaires, j’utiliserai la régression linéaire pour ajuster les données bidimensionnelles. Afin de tracer les données en 2 dimensions, une certaine réduction de la dimensionnalité est nécessaire (le processus de réduction du nombre d’entités en obtenant un ensemble de variables principales). Je vais donner un exemple en utilisant l’algorithme PCA pour résumer les données en 2 variables obtenues avec des combinaisons linéaires des caractéristiques.

Conclusion

Cet article a été un didacticiel pour montrer comment aborder un cas d’usage de régression avec la science des données. J’ai utilisé le jeu de données sur les prix des maisons comme exemple, en passant par chaque étape de l’analyse des données au modèle d’apprentissage automatique.

Dans la section exploratoire, j’ai analysé le cas d’une seule variable catégorielle, d’une seule variable numérique et la façon dont ils interagissent ensemble. J’ai donné un exemple d’ingénierie de fonctionnalité extrayant une fonctionnalité à partir de données brutes. En ce qui concerne le prétraitement, j’ai expliqué comment gérer les valeurs manquantes et les données catégorielles. J’ai montré différentes façons de sélectionner les bonnes fonctionnalités, comment les utiliser pour créer un modèle de régression et comment évaluer les performances. Dans la dernière section, j’ai donné quelques suggestions sur la façon d’améliorer l’explicabilité de votre modèle d’apprentissage automatique.

Une remarque importante est que je n’ai pas couvert ce qui se passe après l’approbation de votre modèle pour le déploiement. Gardez simplement à l’esprit que vous devez créer un pipeline pour traiter automatiquement les nouvelles données que vous obtiendrez périodiquement.

Maintenant que vous savez comment aborder un cas d’utilisation de science des données, vous pouvez appliquer ce code et cette méthode à tout type de problème de régression, effectuer votre propre analyse, créer votre propre modèle et même l’expliquer.
